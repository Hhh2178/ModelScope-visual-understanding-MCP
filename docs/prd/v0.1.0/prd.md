# ModelScope Vision MCP v0.1.0 PRD

## 1. 背景

Hermes 当前主模型基线是 DeepSeek Flash，后续计划增加 DeepSeek Pro 作为复杂任务可选项。DeepSeek 适合承担文本推理、总结和决策，但当前不应假设它可以稳定直接处理图片输入。

因此需要一个独立 MCP，把图片交给专门视觉理解模型处理，再把高质量视觉识别结果传递给 Hermes 主模型继续分析。用户计划优先接入魔搭 ModelScope，因为平台有多个小尺寸视觉模型，并提供每日免费额度。

## 2. 项目目标

v0.1.0 目标是实现一个可被 Hermes 调用的 ModelScope 视觉识别 MCP，先只解决单张图片识别。

成功标准：

- Hermes 可以通过 MCP 调用单张图片识别能力。
- MCP 可以从配置中读取多个魔搭视觉模型。
- 每个模型有可修改的每日调用额度。
- 调用前自动检查额度，额度不足时切换到下一个模型。
- 模型调用失败时自动切换到下一个可用模型。
- 输出不是简单 caption，而是详细、结构化、适合 DeepSeek 后续分析的视觉结果。
- 内置提示词能够引导视觉模型输出高密度视觉描述，覆盖摄影、主体、材质、构图、光影、色彩、文字和不确定性。

## 3. 用户故事

### 3.1 普通图像理解

作为 Hermes 用户，我发送一张图片，希望 Hermes 能先看懂图片内容，再基于图片内容回答问题，而不是只给一句粗略描述。

验收标准：

- 输出包含图片概括、主体、环境、构图、光影、色彩、细节、图片文字和不确定点。
- DeepSeek 接收到的是整理后的视觉证据，不是模型原始废话。

### 3.2 额度自动轮换

作为 Hermes 用户，我希望魔搭某个模型当天额度用完后，MCP 自动切换到下一个模型，而不是直接失败。

验收标准：

- 每个模型配置 `daily_limit`。
- 每次成功调用后计数增加。
- 当 `used_today >= daily_limit` 时跳过该模型。
- 次日自动重置计数。
- 用户可以后续修改模型额度和启用状态。

### 3.3 调用失败 fallback

作为 Hermes 用户，我希望某个视觉模型临时失败、超时或不支持当前图片时，MCP 自动尝试备用模型。

验收标准：

- 返回结果中记录最终使用的模型。
- 如果发生 fallback，返回 fallback 路径和失败原因摘要。
- 所有模型失败时，返回清晰错误报告。

## 4. 功能范围

### 4.1 v0.1.0 必须包含

- MCP 工具：`analyze_image`
- 输入支持：本地图片路径、图片 URL、base64 图片数据
- 魔搭 OpenAI-compatible API 接入
- 多模型配置
- 每日额度计数
- 失败自动切换
- 内置详细视觉识别提示词
- 面向 DeepSeek 的标准输出格式

### 4.2 v0.1.0 暂不包含

- 多图对比
- 视频理解
- 目标检测 bbox 输出
- 表格结构化还原
- 飞书图片附件自动下载链路
- 图像生成
- 自动修改运行态 Hermes 配置

## 5. 输入与输出

### 5.1 MCP 输入

```json
{
  "image": "本地路径 / URL / base64",
  "question": "可选，用户针对图片的问题",
  "detail": "medium",
  "output_format": "deepseek",
  "preferred_model": "可选模型 ID"
}
```

`detail` 可选值：

- `medium`：默认，详细但控制长度。
- `high`：更完整的视觉反推，适合图像生成、审美分析和复杂判断。
- `json`：返回固定字段 JSON，适合程序消费。

### 5.2 MCP 输出

```json
{
  "ok": true,
  "model_used": "Qwen/xxx",
  "fallback_used": false,
  "attempts": [
    {
      "model": "Qwen/xxx",
      "status": "success",
      "reason": ""
    }
  ],
  "deepseek_context": "整理后可直接交给 DeepSeek 的视觉识别文本",
  "vision": {
    "summary": "...",
    "camera": ["..."],
    "subject": ["..."],
    "environment": ["..."],
    "composition": ["..."],
    "lighting": ["..."],
    "color": ["..."],
    "texture": ["..."],
    "text_detected": ["..."],
    "uncertainties": ["..."]
  }
}
```

## 6. DeepSeek 传递要求

传给 DeepSeek 的内容必须详细，但不能无控制膨胀。默认目标长度建议为 600-1200 中文字。

必须包含：

- 图片整体概括
- 主体与关键细节
- 场景与空间关系
- 摄影机位、角度、镜头感
- 构图和视觉重心
- 光影、色彩、材质
- 图片中可见文字
- 不确定性和低置信识别

必须避免：

- 只输出一句 caption
- 输出大量无关赞美
- 混入模型思考过程
- 把无法确认的角色名、品牌名、地点当作事实
- 把原始模型输出不加整理地传给 DeepSeek

建议在 `deepseek_context` 末尾加入约束：

```text
请基于以上视觉识别结果继续分析；不要假设未被识别出的画面细节；如需要判断，请区分已确认事实与推测。
```

## 7. 非功能需求

- 稳定性：单模型失败不应导致整体立即失败。
- 可配置性：模型列表、优先级、额度、超时和启用状态可修改。
- 可审计性：每次调用记录模型、计数、fallback 和失败原因。
- 安全性：不在日志中输出魔搭 token。
- 可迁移性：后续可增加其他视觉平台适配器。
- 路径可移植性：项目内部路径全部使用相对路径，不写死 Windows、WSL、Linux 用户目录或运行态 Hermes 绝对路径。

## 8. 风险

- 魔搭不同模型的图像输入格式可能不完全一致。
- 小尺寸模型对复杂图像、文字、人物细节可能识别不稳定。
- 视觉模型可能产生幻觉，需要在输出中保留不确定性字段。
- 每日额度可能不是单纯“调用次数”，平台实际计费规则需要实测校准。

## 9. 需求合理性分析

该需求合理。用户给出的人像反推提示词本质上是“高密度视觉证据抽取规范”，不只是图像生成 prompt。把它改造成 MCP 内置提示词，可以显著提升视觉模型输出质量，并让 DeepSeek 获得更稳定的上下文。

需要调整的地方是：原提示词偏人像摄影和图像生成反推，而 MCP 应该支持通用图像。因此 v0.1.0 内置提示词会保留摄影、材质、光影、构图等强项，同时增加通用主体、环境、可见文字、用途判断和不确定性约束。
